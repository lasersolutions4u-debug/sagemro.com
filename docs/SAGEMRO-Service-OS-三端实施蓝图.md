# SAGEMRO Service OS 三端实施蓝图

> 状态：已确认，可作为后续开发与验收依据
> 更新时间：2026-06-07

## 1. 产品结构

SAGEMRO Service OS 不是三套割裂的网站，而是一套统一服务系统下的三类角色平台。

| 角色 | 产品名称 | 入口 | 核心定位 |
| --- | --- | --- | --- |
| 客户 / 访客 | SAGEMRO Service OS | `sagemro.com` / `sagemro.cn` | AI 优先沟通、设备档案、服务申请、服务进度 |
| 工程师 / 区域负责人 | SAGEMRO 内部工程师工作台 / Service Representative Console | `engineer.sagemro.com` / `engineer.sagemro.cn` | 区域负责人二次分配、工程师执行派工、查看 AI 初诊摘要、检查清单、填写服务报告 |
| 管理员 / 运营 | SAGEMRO Operations Console | `admin.sagemro.com` / `admin.sagemro.cn` | 线索分流、官方服务确认、派工、报价审核、质量归档 |

## 2. 核心原则

- SAGEMRO 是官方服务体系，不是开放撮合平台，也不是工程师抢单平台。
- 工程师账号和区域负责人账号必须由 SAGEMRO 在确认合作关系后创建和分配，不开放自助注册。
- 工程师端直接独立为 `engineer.sagemro.com` / `engineer.sagemro.cn`，不从主站登录进入。
- 主站只服务客户和访客，避免主站承担工程师作业入口，保持客户体验简单。
- 客户和工程师通过同一个工单会话沟通，但入口、身份和权限分离：客户在主站确认服务进度，工程师在工程师工作台处理任务，Admin 可监管完整记录。
- 客户端保持极简 AI 对话体验，不让用户先选择复杂工具或填写长表单。
- 6 类 AI 能力作为聊天中的隐藏业务结果存在：故障诊断、切割参数、备件识别、维修估算、新机选型、设备健康报告。
- 管理员后台和工程师工作台必须分开。管理员管理系统和服务流程，工程师执行被派发的服务任务。
- `.com` 默认面向国际市场，优先英文；`.cn` 默认面向中国市场，优先简体中文并符合国内合规口径。

## 3. 客户端范围

### 3.1 需要修改

- 删除或隐藏“服务商平台”“工程师抢单”“合伙人接单”“钱包”“抽佣”“提现”等旧平台语言。
- 首页和聊天保持极简，突出“描述问题即可，SAGEMRO AI 帮你整理和推进”。
- 主站只允许客户登录和访客访问；工程师登录入口迁移到独立工程师平台。
- 中文站 `.cn` 后续需要完整中文化，不只是切换 API。
- 英文站 `.com` 默认英文，并按国际客户信任感优化文案。

### 3.2 需要开发

- AI 在聊天中自动整理“服务跟进摘要”。
- 客户确认摘要后，再创建正式服务申请或 CRM 线索。
- 访客聊天可以转成线索，保留来源、AI 摘要、风险等级和推荐下一步。
- 客户可查看设备档案、服务申请、报价确认、服务报告、服务进度和评价。
- 客户可在工单内与 SAGEMRO 官方服务团队和被派工工程师沟通，但客户不能选择工程师或直接发起派单。

## 4. 工程师工作台范围

### 4.1 产品命名

中文：**SAGEMRO 内部工程师工作台**

英文：**Service Representative Console**

不建议使用：

- 合伙人管理台
- 服务商接单中心
- 工程师抢单大厅

### 4.2 需要修改

- 关闭工程师自助注册。
- 区域负责人只看到 Admin 分配给自己区域或团队的线索/服务任务。
- 工程师只看到区域负责人分配给自己的服务任务，不看到全平台客户池或线索池。
- 删除或隐藏工程师端的钱包、抽佣、提现、自由接单、抢单大厅等旧模型。
- 工程师状态改为“派工可用状态”：可派工、暂停派工、离线、停用。

### 4.3 需要开发

- 今日派工。
- 区域待分配任务（仅区域负责人）。
- 待确认派工。
- 服务中任务。
- 待填写服务报告。
- 备件需求。
- 客户设备档案。
- AI 初诊摘要。
- 服务标准检查清单。
- 到场记录、现场照片、处理动作、备件更换、后续建议。
- 首次登录强制修改密码。
- 工程师确认服务规范、安全规范、保密要求。
- 工程师可在被分配的工单内与客户沟通服务细节、现场准备和问题确认，但最终报价、服务结论和归档必须经过 SAGEMRO 官方流程确认。

### 4.4 工程师端角色层级

工程师端至少包含两类账号：

| 角色 | 权限 |
| --- | --- |
| 区域负责人 | 接收 Admin 分配的线索/服务任务；查看区域内工程师能力、负载和状态；将任务分配给具体工程师；跟踪区域任务进度 |
| 工程师 | 查看自己被分配的任务；确认派工；查看 AI 初诊、客户设备档案、备件需求；填写服务报告 |

派工主流程：

1. 客户 / AI 产生线索或服务申请。
2. Admin 在 Operations Console 审核并分配给区域负责人。
3. 区域负责人在 Engineer Console 中查看任务、判断区域资源，并分配给工程师 1、2、3、4、5 等具体人员。
4. 工程师执行服务并提交服务报告。
5. 区域负责人和 Admin 可跟踪结果，必要时审核或归档。

### 4.5 工单沟通和防自派单

工单沟通原则：

- 工单会话属于 SAGEMRO 官方服务记录，不是客户和工程师的私人聊天。
- 客户端、工程师端、管理端共用同一条工单会话数据，但展示界面不同。
- 客户看到服务进度、工程师消息、报价确认、服务报告和完成确认。
- 工程师看到客户问题、AI 初诊摘要、安全风险、设备档案、服务任务、备件准备和客户回复。
- Admin 和区域负责人可查看完整沟通记录，用于服务质量、报价审核和合规归档。

防自派单和利益冲突规则：

- 工程师不得接收自己创建、自己关联公司创建、自己关联手机号/邮箱/地址/设备创建的工单。
- 客户账号不能选择工程师，不能向指定工程师直接派单。
- 工程师不能从线索池主动领取工单，所有派工必须走 Admin -> 区域负责人 -> 工程师。
- 区域负责人不能把存在本人利益冲突的工单分配给自己或关联工程师。
- 报价必须由 Admin 或授权运营人员审核后发送给客户，工程师只能提交现场建议、备件需求和维修方案。
- 客户确认应通过客户账号、短信验证码、邮件确认或其他独立凭证完成，不能只依赖工程师单方面填写。
- 所有分配、改派、报价、客户确认、服务报告提交动作都必须记录审计日志。

- 线索分流。
- 官方服务确认。
- 派工管理。
- 报价管理。
- 备件和新机线索转化。
- 服务质量和合规归档。

### 5.2 需要修改

- 保留现有基础页面：运营驾驶舱、客户与工程师、派工与服务质量、评价管理、线索池。
- 信息架构围绕“线索、审核、派工、报价、服务报告、归档”重新组织。

### 5.3 需要开发

后台首页指标：

- 今日新增 AI 线索。
- 待审核服务申请。
- 高风险停机问题。
- 待报价。
- 待派工。
- 服务中。
- 待归档。
- 备件线索。
- Euchio 新机线索。

线索池：

- 每条线索包含：来源、客户、设备、地区、AI 摘要、风险等级、推荐下一步、分配状态。

派工与服务质量：

## 6. 数据和接口改造

### 6.1 线索表 leads

实施要求：

- `source_type`：`fault_diagnosis_ai`、`cutting_parameter_ai`、`parts_identification_ai`、`repair_estimate_ai`、`machine_selection_ai`、`health_report_ai`、`manual_service_request`、`contact_form`
- `risk_level`：`low`、`medium`、`high`、`critical`
- `ai_summary`
- `recommended_next_step`
- `assignment_status`：`unassigned`、`assigned`、`converted`、`closed`
- `customer_id`
- `device_id`
- `work_order_id`
- `region`

### 6.2 工程师 engineers

实施要求：

- 角色类型：区域负责人 / 工程师。
- 上级区域负责人。
- 合作状态。
- 认证状态。
- 能力标签。
- 服务地区。
- 可服务品牌。
- 当前负载状态。
- 停用状态。

### 6.2.1 区域和团队

实施要求：

- 区域表：区域名称、覆盖城市、省份、负责人。
- 工程师团队归属：工程师归属哪个区域负责人。
- 区域负责人可见范围：只能管理自己区域或团队内的任务和工程师。

### 6.3 服务申请 work_orders

必须支持以下状态流转：

- `pending_review`：待审核。
- `pending_dispatch`：待派工。
- `assigned`：已派工。
- `in_service`：服务中。
- `pending_quote`：待报价。
- `pending_customer_confirmation`：待客户确认。
- `pending_archive`：待归档。
- `completed`：已完成。
- `cancelled`：已取消。

必须补充风控和分配字段：

- `assigned_regional_lead_id`
- `assigned_engineer_id`
- `conflict_status`：`clear`、`review_required`、`blocked`
- `conflict_reason`
- `quote_review_status`
- `customer_confirmation_method`

### 6.3.1 工单会话 work_order_messages

必须新增或规范：

- 关联工单。
- 发送人类型：客户 / 工程师 / 区域负责人 / Admin / 系统。
- 消息内容。
- 附件和图片。
- 是否内部备注。
- 是否客户可见。
- 创建时间。

内部备注必须和客户可见消息分开，避免把报价审核、风控判断、内部派工讨论误发给客户。

### 6.3.2 审计日志 audit_logs

必须新增或规范：

- 操作人。
- 操作对象。
- 操作类型。
- 操作前状态。
- 操作后状态。
- IP / 设备信息。
- 创建时间。

重点记录：账号创建、权限变更、工单分配、改派、报价提交、报价审核、客户确认、服务报告提交、归档。

### 6.4 服务报告 service_reports

必须新增或强化：

- 诊断结论。
- 安全检查清单。
- 处理动作。
- 使用备件。
- 现场照片。
- 工时。
- 客户确认。
- 后续建议。

### 6.5 备件需求 parts_requests

必须新增或强化：

- 备件名称 / 型号。
- 数量。
- 紧急程度。
- 关联设备。
- 关联服务申请。
- 采购状态。

## 7. 执行顺序

1. 保存本蓝图作为正式实施依据。
2. 将管理后台升级为 SAGEMRO Operations Console，并改造后台首页指标。
3. 将 CRM 商机升级为线索池。
4. 将工程师端独立为 `engineer.sagemro.com` / `engineer.sagemro.cn`。
5. 主站只保留客户登录和访客访问。
6. 增加 Admin -> 区域负责人 -> 工程师 的两级分配流程。
7. 增加共享工单会话，让客户和工程师在同一服务记录中沟通。
8. 增加利益冲突识别、禁止自派单、报价审核和审计日志。
9. 为线索补充必要字段和兼容接口。
10. 强化派工、报价、服务报告、客户确认流程。
11. 主线稳定后同步 `.cn` 分支和中文文案。
12. 做生产烟测：客户聊天 -> 线索 / 服务申请 -> Admin 审核 -> 分配区域负责人 -> 区域负责人分配工程师 -> 工程师工单沟通 -> Admin 报价审核 -> 工程师报告 -> 客户确认。
