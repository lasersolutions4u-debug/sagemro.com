# 工程师工作台与工单详情结构调整设计规格

> **项目：** SAGEMRO 工程师端首页 / 工单详情第一阶段重构  
> **日期：** 2026-07-05  
> **状态：** 待确认  

---

## 一、目标

将工程师端从“混合信息面板”调整为更清晰的两层工作流：

1. 首页负责工程师个人工作与运营状态总览。
2. 工单详情负责具体客户问题、设备、AI 初诊、安全风险、配件、报价、服务报告、回款与确认。
3. 回款状态以服务进度信息呈现，不上线钱包、提现、自动分账或平台接单收益表达。
4. 继续保持 SAGEMRO 第三方设备服务与服务协调平台口径，避免“官方审核”“抢单”“钱包”等表述。

本阶段优先使用现有 API 和字段完成前端重排，不新增 D1 migration，不改生产数据结构。

### 2.1 本阶段包含

- 工单详情 `详情` tab 重组为工程师工作分区：
  - 客户问题。
  - 设备信息。
  - AI 初诊摘要。
  - 安全风险。
  - 派工信息。

- 不把工单详情从 modal 改为独立路由。

## 三、首页设计

### 3.1 信息职责

首页只回答三个问题：

1. 我现在有哪些服务任务要处理？
2. 哪些任务需要我马上确认、报价、报告、跟进回款？
3. 我的排单状态和团队分配状态如何？

首页不展示具体工单的 AI 初诊、客户设备档案、安全风险细节。这些内容进入工单详情。

### 3.2 指标分组

指标按现有工单 `status` 推导：

| 指标 | 推导规则 |
| --- | --- |
| 今日派工 | `assigned`、`in_progress`、`in_service` |
| 待确认派工 | `pending`、`pending_dispatch`、`assigned` |
| 服务中 | `in_progress`、`in_service` |
| 待报价 | `pricing` 且报价未提交或仍需处理 |
| 待报告 | `resolved`、`pending_review` 或需要工程师提交服务报告的状态 |
| 待客户确认 | `resolved`、`pending_review` |
| 待回款 | `pending_payment`，或报价已确认但付款未完成的状态 |
| 本月完成 | `completed` 且 `completed_at` 在本月；若列表缺少 `completed_at`，第一阶段展示当前列表内完成数 |

待回款第一阶段优先使用工单状态推导。进入详情后，可根据现有 `getWorkOrderPayment()` 的结果显示更准确付款状态。

### 3.3 首页区域

首页建议保留以下区域：

- 顶部身份与状态：工程师名称、可派工 / 暂停派工 / 离线。
- 工作指标：上述指标卡。
- 近期服务任务：按待处理优先级排序，而不是简单原始顺序。
- 排单日历：保留现有 `EngineerAvailabilityCalendar`。
- 工程师工具箱：只放通用动作入口或占位，例如“服务标准检查清单”“物料申请提示”“服务报告规范”，不放具体工单信息。
- 区域负责人分配模块：仅区域负责人可见，沿用现有团队工程师选择与冲突提示。

## 四、工单详情设计

### 4.1 详情 tab 分区

`WorkOrderDetailModal` 的 `详情` tab 保持单 tab，但内部改为结构化分区，避免本轮大改 tab 路由：

1. **任务头部**：工单号、状态、紧急程度、SLA。
2. **客户问题**：原始描述、工单类型、设备类别。
3. **设备信息**：设备类别、品牌/型号等现有可得字段；如果缺少设备详情，则显示“设备信息待补充”。
4. **AI 初诊摘要**：解析 `ai_summary`，展示摘要和匹配能力；无法解析时不显示。
5. **安全风险**：由 `urgency`、AI 摘要或现有字段推导为常规 / 优先 / 高风险提示。
6. **派工信息**：工程师、客户、地区、提交时间。
7. **配件准备**：提示从报价或服务报告中引用物料库；具体选择仍在报价/服务报告面板完成。
8. **回款与确认**：展示报价状态、客户确认状态、付款状态、服务完成状态、运营结算状态。
9. **服务进度**：沿用现有 logs。

### 4.2 回款与确认口径

工程师可见：

- 报价状态：未报价 / 待运营复核 / 待客户确认 / 已确认。
- 客户确认状态：待确认 / 已确认 / 已完成评价。
- 付款状态：待回款 / 已回款 / 暂无付款记录。
- 服务完成状态：服务中 / 待客户确认 / 已完成。
- 运营结算状态：运营确认中 / 已记录 / 暂未开放。

不显示：

- 钱包余额。
- 提现金额。
- 平台抽佣。
- 工程师收益排行。
- 可抢单收益。

## 五、数据与实现约束

### 5.1 现有数据来源

- 首页工单：`getEngineerTickets(engineerId)`。
- 工单详情：`getWorkOrder(workOrderId)`。
- 回款详情：如需要，使用现有 `getWorkOrderPayment(workOrderId)`。
- 报价详情：现有 `PricingPanels` 内部已读取报价；本轮详情页只做状态摘要，不复制完整报价表单。

### 5.2 状态推导原则

前端新增纯函数负责状态推导，避免 JSX 内散落复杂判断：

- `groupEngineerTickets(tickets, isRegionalLead)`。
- `derivePaymentStage(workOrder, payment)`。
- `deriveWorkOrderActionLabel(ticket)`。

这些函数应可单元测试。

## 八、风险与降级

- 如果 `getWorkOrderPayment()` 在工程师端权限不足，详情页第一阶段只用工单状态展示付款状态，并显示“付款记录由运营维护”。
- 如果列表 API 不返回 `completed_at`，本月完成先显示当前列表中的完成数，后续再由后端补统计接口。
- 如果工单详情 modal 在移动端过长，本轮优先使用分区折叠或更紧凑标题，不改为独立页面。
