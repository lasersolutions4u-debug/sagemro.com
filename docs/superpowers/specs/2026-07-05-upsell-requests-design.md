# 增购与改造需求模块设计规格

> **项目：** SAGEMRO 工程师端 / Admin 增购与改造需求第一阶段  
> **日期：** 2026-07-05  
> **状态：** 已确认  

---

## 一、目标

新增独立的 **增购与改造需求** 模块，让工程师在现场服务、回访、巡检或工单处理中发现客户的配件、易损件、周边设备、自动化改造和折弯模具需求时，可以结构化提交给 SAGEMRO Admin。

Admin 收到需求后，安排业务人员跟进选型、报价、成交判断和交付协同；工程师根据需要协助现场条件确认、安装调试和交付闭环。

本模块明确不做整机新机业务，不将 SAGEMRO 定位改成整机设备销售平台。业务边界仍是第三方售后、配件易损件、激光周边设备、自动化周边和折弯相关耗材/模具。

---

## 二、业务范围

### 2.1 包含

- 配件 / 易损件需求。
- 激光周边设备：
  - 混气装置。
  - 除尘装置。
  - 水冷机。
  - 空压机。
  - 制氮机。
- 后道处理设备：
  - 去毛刺机等。
- 自动化改造：
  - 伺服提升机。
  - 机器人第七轴。
  - 桁架等。
- 折弯相关：
  - 折弯模具。
- 其他现场改造、配套、升级或增购需求。

### 2.2 不包含

- 不做整机新机销售线索。
- 不做完整 CRM、销售漏斗、合同、发票或 ERP。
- 不做库存采购系统。
- 不做工程师佣金、钱包、提现或收益排行。
- 不替代现有维修工单；维修服务仍走工单流程。

---

## 三、核心设计原则

1. **独立模块，可选关联工单。**  
   增购与改造需求有自己的生命周期，不应混入维修工单主状态。来自工单现场的需求可以关联 `work_order_id`，没有工单来源的现场回访也能独立提交。

2. **工程师只负责发现和协助。**  
   工程师提交现场需求、补充技术和现场条件信息，并在成交后协助安装调试。报价、成交判断和业务推进由 Admin / 业务人员处理。

3. **第一版先做结构化需求池。**  
   本阶段重点是提交、查看、分配、状态推进和备注，不做复杂订单、合同、库存、财务闭环。

4. **保留后续扩展空间。**  
   数据结构预留业务负责人、报价状态、成交结果、交付协同备注等字段，方便后续接入报价单、附件、通知和统计。

## 四、数据模型

### 4.1 新表

新增 D1 表：`upsell_requests`。

核心字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 需求 ID |
| `market` | `com` / `cn` |
| `source_type` | `work_order` / `engineer_workspace` |
| `work_order_id` | 可空，来自工单详情时自动关联 |
| `customer_id` | 可空，能识别客户时写入 |
| `engineer_id` | 提交工程师 |
| `category` | 需求分类 |
| `title` | 需求标题 |
| `description` | 需求描述 |
| `site_context` | 现场背景、现有设备、痛点或安装条件 |
| `expected_timeline` | 预计时间 |
| `budget_signal` | 预算信号 |
| `contact_name` | 客户联系人 |
| `contact_phone` | 联系电话 |
| `status` | 处理状态 |
| `assigned_sales_owner` | 业务负责人，可空 |
| `admin_note` | Admin 内部备注 |
| `quote_status` | 报价进展 |
| `deal_result` | 成交 / 未成交 / 未定 |
| `handover_note` | 交付协同说明 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

### 4.2 分类枚举

第一版分类：

- `parts_consumables`：配件 / 易损件。
- `laser_peripheral`：激光周边设备。
- `post_processing`：后道处理设备。
- `automation_retrofit`：自动化改造。
- `bending_tooling`：折弯相关。
- `other_retrofit`：其他现场改造需求。

### 4.3 状态枚举

第一版状态流：

| 状态 | 说明 |
| --- | --- |
| `pending_assignment` | 待分配 |
| `sales_following` | 业务跟进中 |
| `quoted` | 已报价 |
| `won` | 已成交 |
| `lost` | 未成交 |
| `delivery_support` | 交付协同中 |
| `completed` | 已完成 |

`lost` 和 `completed` 是关闭态。`won` 后可进入 `delivery_support`，最后进入 `completed`。

## 五、工程师端设计

### 5.1 首页入口

在工程师首页增加 **增购与改造需求** 入口，建议放在工程师工具箱或任务区域附近。

用途：

- 工程师回访、巡检、现场沟通中发现非维修工单来源的需求。
- 提交时 `source_type = engineer_workspace`。
- `work_order_id` 为空。

### 5.2 工单详情入口

在工单详情中增加 **增购与改造需求** 入口。

用途：

- 工程师处理服务任务时发现客户需要周边设备、配件、易损件、自动化改造或折弯模具。
- 提交时 `source_type = work_order`。
- 自动带入 `work_order_id`、工程师、客户、地区和可用工单上下文。

### 5.3 共用表单字段

两个入口共用同一张表单：

- 需求分类。
- 需求标题。
- 客户现场情况。
- 需求描述。
- 预计时间：
  - 立即。
  - 1 个月内。
  - 3 个月内。
  - 暂不明确。
- 预算信号：
  - 明确预算。
  - 询价比较。
  - 尚未确认。
- 联系人。
- 联系电话。
- 补充备注。

### 5.4 工程师可见范围

第一版工程师可查看自己提交的需求记录和处理状态。区域负责人查看团队提交需求可作为后续扩展，不纳入第一版必须范围。

## 六、Admin 端设计

### 6.1 需求池

Admin 新增 **增购需求池** 页面或菜单项，列表展示：

- 创建时间。
- 市场。
- 分类。
- 标题。
- 来源。
- 工程师。
- 客户。
- 状态。
- 业务负责人。
- 最近更新时间。

筛选条件：

- 状态。
- 分类。
- 市场。
- 来源。
- 工程师。
- 业务负责人。

### 6.2 详情与处理动作

Admin 可在详情中：

- 查看需求全文和来源上下文。
- 分配业务负责人。
- 更新状态。
- 更新报价进展。
- 标记成交 / 未成交。
- 标记交付协同中 / 已完成。
- 填写内部备注。
- 填写交付协同说明。

第一版不做单独销售人员账号模型；`assigned_sales_owner` 可先使用文本字段或 Admin 选择字段，后续再升级为正式用户关联。

## 七、API 设计

### 7.1 工程师侧

- `POST /api/upsell-requests`
  - 工程师创建增购与改造需求。
  - 必须校验登录身份为 `engineer`。
  - 如果传入 `work_order_id`，需要校验该工程师有权访问该工单。

- `GET /api/upsell-requests/mine`
  - 工程师查看自己提交的需求。

### 7.2 Admin 侧

- `GET /api/admin/upsell-requests`
  - Admin 查询需求池。

- `GET /api/admin/upsell-requests/:id`
  - Admin 查看单条需求详情。

- `PATCH /api/admin/upsell-requests/:id`
  - Admin 更新状态、负责人、报价进展、成交结果和备注。

### 7.3 权限原则

- 工程师只能创建和查看自己的记录。
- Admin 可查看和更新全部记录。
- 客户第一版不直接查看该模块，避免把内部商机推进状态暴露给客户。

## 八、通知与运营

第一版可先不做实时推送，但需要让 Admin 在需求池中看到 `pending_assignment` 的待处理记录。

如果现有通知体系接入成本较低，可新增 Admin 站内通知：

- 工程师提交新需求后，通知 Admin 有待分配需求。
- Admin 更新状态后，工程师在自己的需求列表中看到状态变化。

通知不是第一版上线的硬性依赖。

## 九、测试与验收

### 9.1 后端测试

新增 Worker 测试覆盖：

- 工程师可创建独立来源需求。
- 工程师可创建关联工单来源需求。
- 工程师不能为无权访问的工单创建需求。
- 工程师只能查看自己提交的需求。
- Admin 可列表查询需求池。
- Admin 可更新状态和负责人。
- 非 Admin 不能调用 Admin 更新接口。

### 9.2 前端测试

新增或更新前端测试：

- 工程师首页展示 **增购与改造需求** 入口。
- 工单详情展示 **增购与改造需求** 入口。
- 表单分类与时间/预算枚举渲染正确。
- 提交成功后显示成功提示并关闭或重置表单。
- Admin 需求池能渲染不同状态和分类。

### 9.3 本地验证

至少运行：

```bash
npm test --prefix worker
npm run lint --prefix frontend
npm run build --prefix frontend
npm run build --prefix admin
git diff --check
```

如新增 D1 migration，部署 Worker 前必须手动对生产 D1 执行 migration，并确认 `_migrations` 已记录新版本。

## 十、上线与风险

### 10.1 上线顺序

1. 新增 migration 和后端 API。
2. 本地测试通过。
3. 手动在生产 D1 执行 migration。
4. 推送部署 Worker、frontend、admin。
5. 执行生产 smoke。

### 10.2 主要风险

- **业务边界混淆。**  
  文案必须使用“增购与改造需求”，避免“新机”“整机销售”等表达。

- **需求池无人处理。**  
  Admin 页面必须突出 `待分配` 状态，避免工程师提交后无人跟进。

- **字段过重影响提交率。**  
  工程师表单保持轻量，第一版只收必要字段。

- **D1 migration 遗漏。**  
  新表必须先在生产 D1 执行 migration，再部署 Worker。

## 十一、第一版完成标准

- 工程师可从首页提交 **增购与改造需求**。
- 工程师可从工单详情提交关联工单的 **增购与改造需求**。
- Admin 可在需求池中查看、筛选、分配和更新状态。
- 工程师可查看自己提交需求的处理状态。
- 所有新增接口有权限校验和测试覆盖。
- 生产部署前已执行 D1 migration。
