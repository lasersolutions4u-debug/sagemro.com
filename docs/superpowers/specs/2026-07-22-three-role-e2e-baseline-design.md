# Three-Role E2E Baseline Design

## Goal

Create a repeatable browser-level test environment for the SAGEMRO customer, engineer, and Admin applications without reading or writing production business data.

## Scope

The baseline covers two connected journeys:

1. Customer registration and service-order lifecycle: registration, service request, Admin assignment, engineer acceptance and pricing, customer quote and payment-method confirmation, Admin payment confirmation, engineer service record, customer rating, and Admin payout record.
2. Engineer onboarding lifecycle: public application, Admin review, account creation, activation-link delivery, password activation, and engineer login.

The first implementation slice establishes the isolated environment and completes the onboarding lifecycle. The service-order lifecycle follows in the same `e2e/` package after the foundation is green.

## Isolation Model

- Run the frontend and Admin with Vite on dedicated local ports.
- Run the API with `wrangler dev` using a dedicated `e2e/.state` persistence directory.
- Initialize local D1 from `worker/schema.sql` before each clean run.
- Use local KV and R2 bindings supplied by Wrangler.
- Set `ENVIRONMENT=development`, a fixed local verification code, local Admin credentials, and an E2E-only mailbox secret through an ignored `.dev.vars` file generated by the runner.
- Never pass `--remote`, production database IDs, production API hosts, or production credentials to E2E scripts.
- Give every test entity a unique `e2e-<run-id>` identity. The default CI run starts from a fresh local state; retained developer state can be reset explicitly.

## Test Mailbox

Engineer activation must exercise the real token-generation and email-building path. In development only, when `E2E_TEST_MODE=true`, the email sender stores the rendered message in local KV instead of invoking Cloudflare Email or Resend.

A test-only mailbox endpoint returns the latest message when all of these conditions hold:

- `ENVIRONMENT === 'development'`
- `E2E_TEST_MODE === 'true'`
- request header `X-E2E-Test-Secret` exactly matches `E2E_TEST_SECRET`

The route returns 404 in every other environment. Production does not expose activation tokens in responses, logs, D1, or KV through this route.

## Package Layout

- `e2e/package.json`: Playwright dependency and test scripts.
- `e2e/playwright.config.mjs`: three local web servers, browser settings, retries, and artifacts.
- `e2e/scripts/prepare-local-env.mjs`: recreates isolated state, writes ignored development secrets, and initializes D1 from the schema snapshot.
- `e2e/tests/engineer-onboarding.spec.mjs`: public application through activated engineer login.
- `e2e/tests/service-order-lifecycle.spec.mjs`: complete customer/Admin/engineer service workflow.
- `e2e/support/`: API helpers, unique test identities, and role-specific authentication utilities.

## CI

Add a separate E2E job after the existing unit/build job. It installs the browser once, starts all local services through Playwright `webServer`, and uploads trace/screenshots only on failure. Deployment jobs depend on both unit/build verification and E2E once the complete service-order scenario is stable.

During the initial onboarding slice, the E2E job may run independently without gating deployment. It becomes a deployment dependency only after both required journeys pass consistently.

## Failure Reporting

- Each business transition is a separate Playwright step.
- Assertions include the work-order number, application identity, or expected status.
- Traces are retained on first retry and screenshots on failure.
- Local service logs are preserved in CI when startup or API calls fail.

## Security Invariants

- The test mailbox is unreachable in production even if a caller knows the route.
- The mailbox requires an explicit shared secret in development.
- Raw activation tokens are never stored in D1 or application logs.
- E2E scripts reject API base URLs outside loopback addresses.
- Generated local secret files and state directories are gitignored.
