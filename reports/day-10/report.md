## 六、Bug #1 深度分析 — 工程师注册静默失败

### 现场

```
注册后 data-testid:
  'new-chat-button'
  'theme-toggle-button'
  'about-xiaozhi-button'
  'sidebar-login-button'        ← 表示未登录
  'register-engineer-button'    ← 表示表单还在
```

两个矛盾元素共存：系统认为用户未登录，但注册提交按钮仍然可见。

### 诊断步骤

通过独立诊断脚本 `diag_eng3.py` 逐步骤排查：

1. 登录弹窗打开 ✅ → 切换到注册模式
2. 注册表单填写（公司/姓名/密码×2/手机号）✅
3. SMS 验证码请求 → API 监控未捕获响应
4. 身份选择（我是工程师）✅
5. 背景调查表单（专长标签/品牌/地区/自我介绍）填写 ✅
6. `register-engineer-button` 点击 ✅
7. **最终状态：表单仍在，未注册成功** ❌

### 根因推断

两种可能（指向同一类平台问题）：

| 推断 | 证据 | 可能性 |
|------|------|--------|
| 手机号已注册，后端拒绝但前端静默 | 第二轮执行（第一轮已注册该手机号）；`register_user()` 的"已注册→登录"fallback 未触发（body 中未出现"已注册"/"已存在"关键词） | 高 |
| 背景表单标签选中状态异常，后端校验失败静默 | SMS ⚠️、send-code API 未捕获 | 中 |

**核心问题**：平台对注册失败的处理不完善 — 后端拒绝时前端没有展示可读的错误信息，用户（及自动化脚本）无法判断失败原因。

### 建议修复（给开发团队）

1. **注册失败必须展示明确错误** — 当前静默失败，modal 僵死
2. **已注册手机号应直接跳转登录** — 而非卡在注册表单
3. **提交按钮点击后应有 loading 状态** — 防止重复点击，也让状态可感知
4. 复查背景表单的标签组件选中状态判定逻辑
