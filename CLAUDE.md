## 一、项目概述

### 1.1 是什么

SAGEMRO.com 是一个面向钣金加工行业相关设备后服务市场的静态网站，以 **AI 小智作为绝对核心和中枢大脑**。所有交互——无论是客户还是合伙人——都通过小智来连接。客户与 AI 对话获取技术咨询和提交工单，合伙人与 AI 交互处理工单和服务客户。AI 是两者之间的桥梁。

### 1.2 核心定位

**"小智"是整个系统的唯一入口和核心。**

```
客户 ←→ AI 小智 ←→ 合伙人
          ↑
    （所有人通过 AI 连接）
```

- **客户**：直接与小智对话，获得技术咨询。需要人工时，由小智引导提交工单并匹配合适合伙人。
- **合伙人**：直接与小智交互，查看推荐工单、处理工单、获得客户信息摘要。合伙人通过平台总包收款，平台按等级抽佣。
- **AI 小智**：了解每一位客户的设备情况和历史；了解每一位合伙人的专长、等级和服务记录。智能匹配两者的需求。

### 1.3 技术架构

```
GitHub 仓库（代码托管）
    ↓ push / merge
Cloudflare Pages（官网部署）
Cloudflare Workers（AI API 代理 + 工单后端）
    ↓
Cloudflare D1（工单数据、对话数据）
    ↓
OpenAI API（通过"接口AI"中转）
```

### 1.4 技术栈

| 层级 | 技术选型 |
|------|---------|
| 前端框架 | React 18 + Vite |
| 样式 | Tailwind CSS |
| 图标 | Lucide React |
| 动画 | Framer Motion |
| 路由 | React Router DOM |
| 部署 | Cloudflare Pages |
| 后端 | Cloudflare Workers |
| 数据库 | Cloudflare D1（SQLite） |
| AI | OpenAI API（通过接口AI中转） |

## 二、UI 设计总则

### 2.1 设计理念

以 AI 对话为核心，极简 UI，减少导航按钮。所有功能和交互都围绕聊天窗口展开。

### 2.2 视觉风格

- 参考 ChatGPT 的对话式界面
- 极简主义：留白优先，功能内敛
- 主色：`#1677ff`（品牌蓝）
- 字体：系统默认，中文优先

### 2.3 深色模式配色


## 五、关于"关于我们"的内容设计

**设计原则：不单独做"关于我们"页面**，而是通过 AI 小智来介绍。

### 预设问题（FAQ）

在欢迎页显示的快捷问题卡片，同时也是公司介绍的一部分：

| 快捷问题 | AI 回答内容方向 |
|---------|---------------|
| 激光切割毛刺多怎么办？ | 技术回答，同时也是能力展示 |
| 光纤和CO2激光怎么选？ | 技术回答，同时也是能力展示 |
| 切割头多久保养一次？ | 技术回答，同时也是能力展示 |
| 焊接参数怎么调？ | 技术回答，同时也是能力展示 |
| **你们是什么公司？** | 公司介绍：SAGEMRO，钣金加工行业设备后服务平台 |
| **你们提供什么服务？** | 服务介绍：AI咨询 + 合伙人工单 + 设备知识库 |
| **怎么联系你们？** | 联系方式：公司信息 |

### 关于我们页面的替代方案

用户点击"关于小智"时，弹出工单弹窗或发送一条系统预设消息给 AI：

```
用户点击"关于小智" → 自动发送"你们公司是做什么的？" → AI 回答
```

## 行为准则

### 回答技术问题时
- 用户问设备维护、保养、故障相关的知识性问题时，优先基于你的专业知识直接给出有用的回答，不要动不动就推给工程师。你本身就是一个经验丰富的技术顾问，这是你的核心价值之一。
- 回答要结合用户的实际设备情况。如果上下文中有用户的设备列表（品牌、型号），结合具体情况来回答，比泛泛而谈更有价值。
- 涉及安全风险的操作（高压电气部分、激光光路调整、液压系统带压拆卸、机器人示教区域进入等），必须明确提醒用户注意安全或等待专业工程师处理，不能只说怎么做而忽略风险。
- 故障判断只给方向性建议，表述用"可能是""建议检查"而不是"肯定是""一定要"。复杂故障或你不确定的情况，坦诚说明并建议报修让工程师现场诊断。
- 涉及具体配件价格、维修报价时，不要编造数字，建议用户通过报修工单获取工程师的实际报价。

### 推荐和调度工程师时
- 当用户需要工程师上门服务时，综合考量以下因素进行推荐：故障类型与工程师技术专长的匹配度（最关键）、工程师对特定品牌/型号的经验、地理距离与预计响应时间、工程师当前工单负荷、历史服务评价与客户反馈。
- 推荐时要向用户清晰说明推荐理由，让用户感到你是真正在帮他匹配最合适的人选，而不是随机分配。
- 如果最优人选当前不可用，主动给出替代方案和等待时间预估，让用户自行决策。
- 对于紧急故障（停产级别），优先推荐响应最快且能力匹配的工程师，即使不是评分最高的。

### 处理报修时
- 当用户表达报修意图时，按顺序引导采集：哪台设备出问题 → 什么故障现象 → 紧急程度。如果用户一次性说清楚了，不要重复追问。
- 创建工单前，必须将工单信息汇总展示给用户确认，得到确认后才调用创建工具。
- 如果用户描述的故障你能初步判断，在引导报修的同时附带简短的应急建议。

### 沟通风格
- 用简洁的口语化中文。不要用"您好，很高兴为您服务"这种客服腔。直接、有用、像懂行的同事。
- 如果用户的问题跟设备维修完全无关，可以简单回应但自然地引导回主线。
- 每次回复控制在合理长度。简单问题2-3句话，技术问题可以稍长但也不超过一屏，要有重点。
- 如果用户明显是行家（比如工程师身份或使用了专业术语），可以用更专业的方式交流，不用过度解释基础概念。

### 9.1 D1 数据库 Schema

```sql
-- 客户用户表
CREATE TABLE customers (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    phone TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    region TEXT,
    created_at TEXT DEFAULT (datetime('now'))

-- 合伙人（工程师）表
CREATE TABLE engineers (
    id TEXT PRIMARY KEY,
    user_no TEXT UNIQUE NOT NULL,    -- E-0001
    name TEXT NOT NULL,
    phone TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    salt TEXT NOT NULL,

    -- 背景调查信息（均为自由填写的标签）
    specialties TEXT NOT NULL,       -- JSON数组：设备类型标签 ["激光切割机", "折弯机", "焊接机"]
    brands TEXT,                     -- JSON：按设备类型分类的品牌标签 {"激光切割机": ["大族", "通快"], "折弯机": ["通快", "百超"]}
    services TEXT NOT NULL,           -- JSON数组：维修保养项目标签 ["激光器维修", "切割头维护", "参数调试"]
    service_region TEXT,              -- 服务覆盖地区
    bio TEXT,                         -- 个人简介

    -- 状态和评分
    status TEXT DEFAULT 'available',  -- available / paused / offline
    rating_timeliness REAL DEFAULT 0,
    rating_technical REAL DEFAULT 0,
    rating_communication REAL DEFAULT 0,
    rating_professional REAL DEFAULT 0,
    rating_count INTEGER DEFAULT 0,

    -- V2 合伙人总包制字段
    level TEXT DEFAULT 'junior',       -- junior / senior / expert
    commission_rate REAL DEFAULT 0.80, -- 提成比例：Junior 80% / Senior 85% / Expert 88%
    credit_score INTEGER DEFAULT 100,   -- 信用分（初始100）
    deposit_balance INTEGER DEFAULT 0, -- 保证金余额（元）
    wallet_balance INTEGER DEFAULT 0,  -- 钱包余额（元）
    total_orders INTEGER DEFAULT 0,    -- 累计完成工单数
    complex_orders INTEGER DEFAULT 0, -- 累计完成复杂工单数
    success_orders INTEGER DEFAULT 0, -- 累计成功工单数

    created_at TEXT DEFAULT (datetime('now'))
);

-- 设备表（客户绑定的设备，设备类型和品牌均为标签）

    id TEXT PRIMARY KEY,
    customer_id TEXT,
    type TEXT NOT NULL,        -- 设备类型标签（自由填写）

    model TEXT,                 -- 型号
    power TEXT,                 -- 功率
    created_at TEXT DEFAULT (datetime('now')),
    FOREIGN KEY (customer_id) REFERENCES customers(id)
);

-- 工单表

    type TEXT NOT NULL,         -- fault / maintenance / parameter / other

    status TEXT DEFAULT 'pending',  -- pending / assigned / in_progress / pricing / in_service / resolved / pending_review / completed / rejected / cancelled
```

## 十四、工单消息对话与核价系统 ⭐ 已完成 ✅

### 14.1 核心理念

**交易平台模式下，工单不仅是"报修单"，更是一笔在平台见证下完成的交易。** 合伙人接单后，双方应在平台内完成：核价 → 报价 → 确认 → 服务 → 评价。整个过程平台留有记录，平台按合伙人等级抽取佣金，是商业变现的核心路径。

### 14.2 工单完整流程

```
① 客户 → AI 小智描述问题
     ↓
② AI 判断是否需要人工
     ↓（需要）
③ 客户提交工单 → AI 推荐合伙人 → 合伙人接单
     ↓
④ 【核心阶段】工单消息对话 + 核价
     - 合伙人在工单内发起核价
     - 填写：人工费、配件费、差旅费、其他费用
     - AI 自动审核报价（对比地区历史数据）
     - 客户查看含平台服务费的全包价报价单
     - 客户确认报价（或发起议价对话）
     ↓
⑤ 客户确认报价 → 工单进入"服务中"（in_service）
     ↓
⑥ 合伙人上门服务 → 标记完成 → 客户评价 → 工单完成
```

### 14.3 核价模块（面向合伙人）

合伙人在工单详情页填写核价单：

| 费用项 | 说明 |
|--------|------|
| 人工费 | 工时（小时）× 单价（元/小时）|
| 配件费 | 配件名称 + 单价 + 数量 |
| 差旅费 | 交通费 + 住宿费 |
| 其他费用 | 其他杂项，可填写说明 |
| 小计 | 人工+配件+差旅+其他 |
| 平台服务费 | 自动计算 = 小计 × (1 - 合伙人提成比例) |
| 动态保证金（5%） | 小计 × 5%，进入风控池 |
| **合伙人实得** | 小计 × 提成比例，进入合伙人钱包 |
| **客户应付** | = 小计（合伙人报的全包价）|

**佣金体系说明（V2 合伙人总包制）：**
- 合伙人注册默认 Junior 级，提成比例 80%（平台收 20%）
- 等级晋升后比例提升：Senior 85% / Expert 88%
- 动态保证金：每单结算时强制扣留 5% 进入风控池（暂缓实施）
- 合伙人提现：申请后 T+1 到账（需绑定银行卡）
- 佣金明细在报价单中清晰展示，双方知情

### 14.4 AI 报价审核

合伙人提交报价后，AI 自动分析并给出判断：

| AI 判断 | 说明 |
|---------|------|
| 合理 ✅ | 价格在地区平均水平区间内 |
| 偏高 ⚠️ | 价格高于地区平均 X%，说明偏高原因 |
| 偏低 💡 | 价格低于地区平均 X%，建议适当上调 |

AI 审核依据：
- 该合伙人历史报价记录
- 同地区同类工单的平均报价
- 设备类型和故障复杂度

### 14.5 客户报价确认

客户收到报价后：
- 查看完整明细（含佣金拆分）
- 查看 AI 价格审核结论
- 选择：**确认报价** 或 **发起议价**
- 议价时可在工单消息区继续对话

### 14.6 工单消息区

工单内的实时对话区，客户和合伙人均可用。

**消息类型：**
- `text` — 普通文字消息
- `pricing_update` — 报价更新通知（系统消息）
- `system` — 系统通知（如"客户已确认报价"）

**功能要求：**
- 工单详情页内嵌消息列表
- 登录用户均可发送消息
- 新消息实时刷新
- 消息记录长期保存，是服务过程的凭证

### 14.7 状态变更

工单状态补充：

|------|------|

## 十五、AI 全数据访问（系统）⭐ 重点处理

### 15.1 为什么需要 AI 访问所有数据

当前 AI 小智的 system prompt 包含大量知识，但**它不知道平台上的真实用户数据**：
- 不知道具体客户有哪些设备、历史工单
- 不知道合伙人的真实等级、评分和接单状态
- 不知道当前工单的进展

这导致 AI 只能泛泛而谈，无法给出"针对你这个客户"的精准建议。

**我们要把平台变成一个有记忆的 AI**，让小智认识每一个客户、每一个合伙人，随着数据积累越来越懂他们。

### 15.2 AI Function Calling 工具

通过 OpenAI Function Calling，让 AI 按需查询数据库：

| 工具名 | 功能 |
|--------|------|
| `get_customer_profile` | 查询客户档案：设备列表、品牌型号、历史工单、评价记录 |
| `get_engineer_profile` | 查询合伙人档案：专长、等级、提成比例、信用分、评分、接单状态、历史工单 |
| `get_work_order_detail` | 查询工单详情：当前状态、历史消息、报价记录 |
| `get_pricing_history` | 查询历史报价数据（用于 AI 核价参考） |
| `get_regional_pricing` | 查询某地区某设备类型工单的平均报价 |
| `get_conversation_history` | 查询用户最近 N 条对话历史 |

### 15.3 长期记忆与数据积累

**这是我们平台的核心竞争壁垒：数据越积累，AI 越聪明。**

积累路径：
1. **客户数据积累**：每台设备、每次故障、每次维修记录 → AI 了解设备全貌
2. **合伙人数据积累**：每次报价、每次服务、每个评价 → AI 了解合伙人真实能力（含等级晋升路径）
3. **地区价格数据积累**：各地区、各设备类型、各故障类型的维修价格 → AI 学会定价

**数据驱动 AI，AI 反哺平台：**
- AI 根据客户设备历史主动推荐保养计划（预防性维护）
- AI 根据合伙人历史报价给出定价建议（按合伙人等级参考同类报价）
- AI 识别异常报价（远高于或远低于市场均价）
- AI 匹配最优合伙人（不只是专长，还看价格、距离、评价综合考量）

### 15.4 AI System Prompt 扩展方向

在现有 system prompt 基础上，新增指令：

**关于客户：**
> 你了解每一位客户的设备档案。当客户提到设备问题时，结合该客户的设备历史（如果有）给出更有针对性的回答。如果检测到设备反复出现同类故障，主动建议预防性保养。

**关于合伙人报价：**
> 当合伙人询问如何报价时，调用 `get_regional_pricing` 工具，参考同地区同类工单的平均价格给出建议，并结合合伙人等级给出合理区间。当客户询问报价是否合理时，基于历史数据给出判断。

**关于客户主动关怀：**
> 根据客户设备的上次保养时间和使用频率，在合适时机主动提醒保养。

## 十六、平台优势与战略亮点

> 以下内容由 AI 基于对平台定位和功能设计的分析提炼，帮助团队理解"我们为什么做这个平台"。

### 我们解决什么问题？

钣金加工行业的设备售后服务，长期存在三大痛点：

1. **找合伙人难**：不知道去哪里找靠谱的合伙人，品牌代理商太贵，社会维修师傅不放心
2. **价格不透明**：维修费用没有标准，讨价还价费时费力，经常被宰
3. **质量无保障**：修完没保障，出了问题找不到人，没有记录可查

### 我们的解决方案

**小智是入口，平台是保障，AI 是大脑。**

```
客户 → 小智（AI咨询，判断是否需要合伙人）
     ↓（需要）
     平台匹配 → 合伙人接单 → 工单内核价/议价
     ↓（价格透明，双方确认）
     上门服务 → 平台记录 → 评价 → 完成
```

### 我们的核心优势

| 维度 | 传统方式 | 我们 |
|------|---------|------|
| 找合伙人 | 口碑推荐/代理商 | AI 精准匹配最优合伙人 |
| 定价 | 口头议价，不透明 | 平台核价，AI 审核，公开透明 |
| 服务保障 | 无记录，出了问题找不到人 | 工单全程记录，平台托管 |
| 知识积累 | 无 | AI 记住每一台设备、每一次维修 |
| 数据价值 | 无 | 积累形成地区价格标准、AI 定价能力 |

### 为什么 AI 是壁垒？

**别人可以抄界面，但抄不走我们的数据。**

- 每一次工单完成，都是一次数据积累
- 每一次报价确认，都是市场定价的学习
- 每一次评价反馈，都是合伙人能力的验证

**数据越多，AI 越准 → AI 越准，用户越信任 → 用户越多，数据越多**

这是一个正向飞轮，是平台的竞争壁垒。

### 我们对用户的价值

**对客户：**
- 不再四处打听哪里有靠谱的合伙人
- 报价透明，心里有底，不被宰
- 有平台记录，服务有保障

**对合伙人：**
- 稳定的订单来源（平台派单）
- 不用自己谈价格（平台定价参考）
- 建立个人品牌（评价体系 + 等级晋升）
- AI 辅助报价，不担心开价不合理

```
Prompt AI（当前） → Function Calling（进行中） → RAG（规划） → Agent AI（远期）
```

**已完成工具：**
- `get_engineer_profile` — 查询合伙人档案（等级/钱包/评分/工单数/本月收入）

### 18.4 第三阶段：RAG（规划中，2-4周）

**核心转变：** 从"每次注入上下文"到"AI 主动检索知识库"

**RAG 数据来源（不需要人工整理文档）：**

| 数据源 | 内容 | 获取方式 |
|--------|------|---------|
| 历史工单 | 故障描述 + 维修方案 + 实际报价 | 从 D1 `work_orders` 表导出 |
| 对话记录 | 客户问什么问题，AI 怎么回答 | 从 D1 `messages` 表导出 |
| 维修记录 | 每次维修的解决方案 | 工单完成后由合伙人填写 |
| 报价记录 | 地区/设备/故障类型的成交价 | 从 D1 核价相关表导出 |
| 评价内容 | 客户反馈文字 | 从 D1 `ratings` 表导出 |

**技术方案：**
- 不需要购买向量数据库（如 Pinecone）
- 使用 D1 FTS5 全文索引（已有 SQLite 支持）
- 或使用 Cloudflare Search（轻量级搜索服务）

**效果：**
- 客户问"之前有过类似问题吗"→ AI 从历史工单检索相似案例
- 合伙人问"这类故障一般怎么报价"→ AI 从历史报价记录给出参考
- AI 报价审核自动对比历史数据，不再拍脑袋

### 18.5 第四阶段：Agent AI（远期，3-6个月）

**核心转变：** AI 不只是回答问题，而是能**规划路径、多步执行、接管工作流**

**Agent 场景示例：**

```
场景1：客户说"我的激光切割机坏了"
Agent 行为：
  Step 1 → 查客户档案，找到这台设备
  Step 2 → 查历史工单，看这类故障之前怎么处理过
  Step 3 → 判断是否需要上门（80% AI 能直接解决）
  Step 4 → 如果需要，调用 get_matching_engineers，列出最优 3 人
  Step 5 → 把推荐结果呈现给客户，说明理由
  Step 6 → 客户确认后，创建工单，发送推送

场景2：合伙人报价后
Agent 行为：
  Step 1 → 调用 get_regional_pricing，对比地区均价
  Step 2 → 调用 get_engineer_history，查该合伙人历史报价记录
  Step 3 → 生成审核结论：合理/偏高/偏低，给出建议
  Step 4 → 将审核结果写入工单消息区，通知客户

场景3：设备预防性保养
Agent 行为：
  - 根据客户设备的上次保养时间，超过 6 个月时主动提醒
  - 结合设备使用频率（工单数量）综合判断保养优先级
```

**Agent 需要的前提条件（正在积累）：**
- ✅ 足够多的历史工单数据（定价依据）— 进行中
- ✅ 足够多的客户设备档案（匹配依据）— 本阶段实现
- ✅ 足够多的合伙人信息（推荐依据）— 已完成
- ✅ 评价体系（质量验证）— 已完成

### 18.6 知识库建设路径

**你们的独特优势：不需要人工整理文档**

传统 RAG 需要：产品手册 + 技术文档 + 规范文档（大量人工投入）
你们的 RAG 可以有：工单数据 + 对话记录 + 报价历史 + 评价内容（随业务自然增长）

**数据飞轮：**
```
每次工单完成 → 维修记录沉淀 → RAG 知识增长
每次报价确认 → 价格数据积累 → AI 定价能力
每次评价反馈 → 合伙人能力验证 → 推荐更精准
```
